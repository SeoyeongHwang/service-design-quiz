# PRD: 서비스·경험디자인 기사 필기 퀴즈 MVP

## 1. 제품 개요

**제품명:** 서비스·경험디자인 기사 필기 퀴즈 MVP  
**제품 유형:** 모바일 우선 정적 웹 퀴즈 앱  
**목표:** 사용자가 모바일 웹에서 서비스·경험디자인 기사 필기시험의 4개 섹션 중 하나를 선택하고, 섹션 내 문제를 중복 없이 랜덤으로 풀면서 즉시 정답과 해설을 확인할 수 있도록 한다.

서비스·경험디자인 기사 필기시험은 객관식 80문항, 2시간 30분 형식이며, 4개 필기과목이 각각 20문항으로 구성된다. MVP는 이 시험 구조를 반영하여 4개 섹션 단위로 문제를 선택하고 풀 수 있게 한다.

---

## 2. MVP 핵심 방향

MVP는 “학습 기록 관리 앱”이 아니라 **가볍게 접속해서 섹션별 문제를 끝까지 푸는 모바일 퀴즈 앱**으로 정의한다.

| 구분 | MVP 결정 |
|---|---|
| 로그인 | 없음 |
| DB | 없음 |
| 결과 저장 | 없음 |
| 문제 출제 | 미리 만든 문제 더미에서 섹션별 중복 없는 랜덤 출제 |
| 문제 형식 | 4지선다 또는 5지선다 객관식 |
| 해설 | 정답/오답 이유 + 이론서 페이지 |
| 사용 환경 | 모바일 웹 우선 |
| 배포 | GitHub Pages 기반 정적 웹앱 |

---

## 3. 문제 생성 방식

### 3.1 결론

**MVP에서는 문제를 실시간 생성하지 않고, 사전에 생성한 문제 더미를 사용하는 것이 적합하다.**

시험 대비 퀴즈에서 중요한 것은 재미보다 **정답 신뢰도, 해설 신뢰도, 이론서 페이지 연결 정확도**이다. 실시간 생성 방식은 그럴듯한 오답, 부정확한 페이지 매칭, 애매한 정답 근거가 생길 수 있으므로 MVP 품질을 흔들 가능성이 있다.

### 3.2 문제 더미 준비 방식

문제 더미는 다음 방식으로 준비한다.

1. 이론서, 핵심키워드 핸드북, 출제기준을 기준으로 문제를 생성한다.
2. 각 문제에 정답, 해설, 관련 페이지, 출제 섹션, 키워드를 붙인다.
3. 앱은 이 문제 더미에서 섹션별로 이미 푼 문제를 제외하고 랜덤 출제한다.

### 3.3 저작권 및 데이터 사용 원칙

“기출문제 같은 형식”은 참고하되, **실제 기출문제를 그대로 복제해서 공개 웹사이트에 올리는 방식은 피한다.** MVP 문제 데이터는 이론서 기반의 변형 문제 또는 자체 제작 문제로 구성한다.

## 4. 사용자 시나리오

### 4.1 기본 시나리오

1. 사용자가 모바일 브라우저로 웹사이트에 접속한다.
2. 홈 화면에서 필기시험의 4개 섹션 중 하나를 선택한다.
3. “문제 풀기 시작하기” 버튼을 누른다.
4. 문제 풀이 화면으로 이동한다.
5. 문제 카드에 객관식 문제가 표시된다.
6. 사용자가 보기 중 하나를 선택한다.
7. 사용자가 “정답 확인” 버튼을 누른다.
8. 정답 결과가 표시된다.
9. 결과 화면에서 정답 여부, 정답/오답 이유, 이론서 페이지를 확인한다.
10. “다음 문제” 버튼을 누르면 같은 섹션의 아직 풀지 않은 다음 문제가 나온다.
11. 섹션 내 모든 문제를 풀면 완료 메시지가 표시된다.

## 5. 정보 구조

### 5.1 화면 목록

| 화면 | 목적 |
|---|---|
| Home | 앱 소개, 4개 섹션 선택 |
| Section Detail | 선택한 섹션 설명, 시작 버튼 |
| Quiz | 문제 카드, 보기 선택 |
| Answer Result | 정답 여부, 해설, 관련 페이지 |
| Complete | 섹션 내 모든 문제 풀이 완료 안내 |
| Empty/Error | 문제 없음, 데이터 오류 대응 |

MVP에서는 상세한 점수 요약 페이지를 만들지 않는다. 섹션 완료 여부는 안내하되, 회차 단위 통계보다 **문항 단위 피드백**이 더 중요하다.

## 6. 섹션 구조

필기시험의 4개 섹션은 다음과 같이 앱에 반영한다.

| 앱 표시명 | 내부 ID |
|---|---|
| 서비스·경험디자인 기획설계 | `planning` |
| 사용자 조사·분석 | `research` |
| 사용자 중심 전략수립 | `strategy` |
| 서비스·경험디자인 개발 및 운영 | `development_operation` |

홈 화면에서는 4개 섹션을 카드형으로 보여준다. 각 카드에는 섹션명, 짧은 설명, 예상 문제 범위를 표시한다.

### 6.1 섹션 카드 예시

**서비스·경험디자인 기획설계**  
디자인 개요, 디자인씽킹, 서비스디자인, 경험디자인, UI, 사용성 평가, 요구사항 파악 등을 다룹니다.

## 7. 핵심 기능 요구사항

## F1. 섹션 선택

사용자는 홈 화면에서 4개 섹션 중 하나를 선택할 수 있어야 한다.

### 수용 기준

| 조건 | 기대 결과 |
|---|---|
| 사용자가 섹션 카드를 누름 | 섹션 상세 또는 바로 시작 화면으로 이동 |
| 섹션에 문제가 있음 | “문제 풀기 시작하기” 버튼 활성화 |
| 섹션에 문제가 없음 | “아직 준비된 문제가 없습니다” 표시 |

---

## F2. 문제 풀이 시작

사용자가 “문제 풀기 시작하기”를 누르면 선택한 섹션의 아직 풀지 않은 문제 중 하나가 랜덤으로 표시된다.

### 수용 기준

| 조건 | 기대 결과 |
|---|---|
| 문제 더미에 해당 섹션 문제가 있음 | Quiz 화면 진입 |
| 문제 더미가 비어 있음 | Empty 화면 표시 |
| 이전에 푼 문제가 있음 | 같은 세션 내에서는 해당 문제를 다시 출제하지 않음 |

---

## F3. 문제 카드 표시

문제 풀이 화면은 하나의 문제를 하나의 카드로 표시한다.

### 카드 포함 정보

| 요소 | 필수 여부 |
|---|---|
| 섹션명 | 필수 |
| 문제 번호 또는 임시 카운트 | 선택 |
| 문제 문항 | 필수 |
| 보기 4개 또는 5개 | 필수 |
| 선택 상태 | 필수 |
| 정답 확인 버튼 | 필수 |
| 난이도 | 선택 |
| 키워드 | 선택 |

MVP에서는 스와이프보다 **정답 확인 버튼을 우선 구현**한다. 스와이프는 모바일 경험상 매력적이지만, 실수로 넘김, 접근성 문제, 구현 시간 증가가 생길 수 있다. 버튼 기반 플로우를 먼저 만들고, 이후에 스와이프 제스처를 추가한다.

---

## F4. 보기 선택

사용자는 보기 중 하나만 선택할 수 있다.

### 수용 기준

| 조건 | 기대 결과 |
|---|---|
| 사용자가 보기를 누름 | 해당 보기가 선택 상태로 표시 |
| 다른 보기를 다시 누름 | 선택이 변경됨 |
| 선택 전 정답 확인 버튼 | 비활성화 |
| 선택 후 정답 확인 버튼 | 활성화 |

---

## F5. 정답 결과 표시

사용자가 정답 확인 버튼을 누르면 Answer Result 화면으로 이동하거나, 같은 화면에서 카드가 결과 모드로 전환된다.

**권장 구현:** 같은 화면에서 결과 모드로 전환한다.  
이유는 화면 이동이 적고, 모바일에서 퀴즈 흐름이 더 빠르기 때문이다.

### 결과 화면 포함 정보

| 요소 | 필수 여부 |
|---|---|
| 정답/오답 상태 | 필수 |
| 사용자가 선택한 답 | 필수 |
| 정답 | 필수 |
| 정답인 이유 | 필수 |
| 오답 선택지가 틀린 이유 | 권장 |
| 관련 이론서 페이지 | 필수 |
| 관련 키워드 | 권장 |
| 다음 문제 버튼 | 필수 |

---

## F6. 다음 문제

사용자가 “다음 문제”를 누르면 같은 섹션 내에서 아직 풀지 않은 다음 문제가 랜덤으로 표시된다.

### 수용 기준

| 조건 | 기대 결과 |
|---|---|
| 다음 문제 버튼 클릭 | 아직 풀지 않은 새 문제 카드 표시 |
| 섹션의 모든 문제를 품 | 완료 메시지 표시 |
| 다시 풀기 클릭 | 같은 섹션 문제를 새 랜덤 순서로 다시 시작 |
| 페이지 새로고침 | 세션 상태 초기화 가능 |

---

## F7. 저장하지 않기

MVP에서는 퀴즈 결과를 localStorage, 서버, DB에 저장하지 않는다.

다만 앱 내부 상태로는 현재 문항의 선택값과 정답 확인 여부를 유지해야 한다. 새로고침하면 초기화되어도 된다.

## 8. 문제 데이터 설계

문제 더미는 정적 파일로 관리한다.

### 8.1 추천 파일 구조

```txt
src/
  constants/
    sections.ts
  data/
    questions.ts
  types/
    question.ts
  utils/
    getRandomQuestion.ts
```

### 8.2 Question 데이터 필드

| 필드 | 설명 | 예시 |
|---|---|---|
| `id` | 고유 ID | `planning-001` |
| `section` | 4개 섹션 ID | `planning` |
| `sourceType` | 출처 유형 | `theory`, `keyword`, `generated` |
| `question` | 문제 문항 | `다음 중 서비스디자인의 특징으로 옳지 않은 것은?` |
| `choices` | 보기 배열 | 4개 또는 5개 |
| `answerIndex` | 정답 보기 index | `2` |
| `explanation` | 정답 해설 | `서비스디자인은 사용자 중심의 맥락적 리서치를 중시한다.` |
| `wrongExplanations` | 오답 해설 배열 | 각 보기가 틀린 이유 |
| `theoryPage` | 이론서 페이지 | `85` |
| `theoryRange` | 페이지 범위 | `85-88` |
| `keywords` | 관련 키워드 | `서비스디자인`, `디자인씽킹` |
| `difficulty` | 난이도 | `low`, `medium`, `high` |
| `examStyle` | 문제 유형 | `definition`, `process`, `comparison`, `application` |

### 8.3 TypeScript 타입 예시

```ts
type SectionId =
  | "planning"
  | "research"
  | "strategy"
  | "development_operation";

type SourceType = "theory" | "keyword" | "generated";

type Difficulty = "low" | "medium" | "high";

type ExamStyle =
  | "definition"
  | "process"
  | "comparison"
  | "application";

export type Question = {
  id: string;
  section: SectionId;
  sourceType: SourceType;
  question: string;
  choices: string[];
  answerIndex: number;
  explanation: string;
  wrongExplanations?: string[];
  theoryPage?: number;
  theoryRange?: string;
  keywords?: string[];
  difficulty?: Difficulty;
  examStyle?: ExamStyle;
};
```

## 9. 문제 더미 생성 원칙

문제는 단순 암기형만 만들지 않고, 최소한 다음 4가지 유형을 섞는다.

| 유형 | 설명 |
|---|---|
| 정의 확인형 | 개념의 정의를 묻는 문제 |
| 구분 비교형 | 유사 개념 간 차이를 묻는 문제 |
| 프로세스 순서형 | 단계, 절차, 흐름을 묻는 문제 |
| 적용 판단형 | 상황을 주고 적절한 방법론을 고르는 문제 |

초기 문제 수는 **섹션당 20문항, 총 80문항**을 목표로 한다. 실제 필기시험도 4개 과목 × 20문항 구조이므로, 1차 문제 더미를 시험 1회분 규모로 맞추는 것이 자연스럽다.

## 10. UX 원칙

### 10.1 모바일 우선

- 한 화면에는 하나의 핵심 행동만 둔다.
- 문제 문항과 보기는 세로 스크롤 없이 최대한 한 카드 안에서 보이게 한다.
- 보기 버튼은 엄지로 누르기 쉽게 충분한 높이를 둔다.
- 정답 확인 버튼은 하단 고정 CTA로 둔다.
- 정답/오답은 색상뿐 아니라 텍스트와 아이콘으로도 구분한다.

### 10.2 정답 확인 흐름

MVP에서는 아래 흐름을 기본으로 한다.

```txt
문제 표시 → 보기 선택 → 정답 확인 → 결과 표시 → 다음 미풀이 문제 또는 섹션 완료
```

| 기능 | MVP 포함 여부 |
|---|---|
| 버튼으로 정답 확인 | 포함 |
| 카드 스와이프 | 제외 또는 선택 |
| 자동 다음 문제 이동 | 제외 |
| 정답 결과 애니메이션 | 간단히 포함 가능 |

## 11. 비기능 요구사항

| 항목 | 요구사항 |
|---|---|

| 오프라인 | MVP 필수 아님 |
| 보안 | 로그인/개인정보 없음 |
| 브라우저 | 최신 Chrome, Safari 모바일 기준 |

## 12. 제외 범위

MVP에서는 아래 기능을 만들지 않는다.

| 제외 기능 | 이유 |
|---|---|
| 로그인 | 불필요한 복잡도 증가 |
| 학습 기록 저장 | MVP 목표와 무관 |
| 오답노트 | v2로 충분 |
| 점수 통계 | MVP에서는 문항 단위 피드백이 우선 |
| AI 실시간 문제 생성 | 정답 신뢰도와 페이지 매칭 리스크 |
| 관리자 페이지 | 문제 수가 적을 때는 정적 데이터 관리가 빠름 |
| 실제 기출문제 원문 수록 | 저작권 리스크 |

## 13. 개발 우선순위

### Sprint 1: 기본 퀴즈 플로우

- 4개 섹션 홈 화면
- 섹션 선택
- 문제 풀기 시작
- 문제 카드 표시
- 보기 선택
- 정답 확인
- 결과 표시
- 다음 문제
- 섹션 완료 안내

### Sprint 2: 문제 데이터 고도화

- 섹션당 20문항 문제 더미 입력
- 이론서 페이지 연결
- 정답/오답 해설 정리
- 난이도와 키워드 태그 추가

### Sprint 3: 모바일 UX 개선

- 카드 UI 다듬기
- 하단 CTA 고정
- 정답/오답 시각 피드백
- 섹션 내 풀이 완료 전 중복 방지
- 간단한 전환 애니메이션

## 14. 성공 기준

MVP가 성공했다고 볼 수 있는 기준은 다음이다.

| 기준 | 목표 |
|---|---|
| 사용자가 섹션을 선택할 수 있음 | 가능 |
| 문제를 풀고 정답을 확인할 수 있음 | 가능 |
| 해설과 이론서 페이지를 볼 수 있음 | 가능 |
| 섹션 내 문제를 중복 없이 끝까지 풀 수 있음 | 가능 |
| 새로고침해도 앱이 깨지지 않음 | 가능 |
| 모바일에서 불편 없이 조작 가능 | 가능 |
| 문제 데이터 추가가 어렵지 않음 | 가능 |

## 16. 최종 MVP 정의

**사용자가 모바일 웹에서 필기시험의 4개 섹션 중 하나를 선택하고, 사전 제작된 이론서 기반 객관식 문제를 카드형으로 중복 없이 랜덤 풀이하며, 매 문항마다 정답 여부와 해설, 관련 이론서 페이지를 즉시 확인할 수 있는 정적 퀴즈 웹앱.**

개발 관점에서 가장 중요한 것은 문제 생성 기능이 아니라 **섹션 선택 → 문제 풀이 → 정답 확인 → 다음 미풀이 문제 → 섹션 완료 안내의 흐름을 매끄럽게 만드는 것**이다. 문제 데이터는 정적 파일로 시작하되, 나중에 DB로 이전하기 쉬운 구조로 설계한다.
